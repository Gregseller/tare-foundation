# 🟦 **ФАЗА 1 — FOUNDATION (БАЗИС)**  
### *Цель: создать фундамент, который нельзя будет переделывать*

## Что создаём:
- **Time Engine** (единая временная модель)  
- **TickDataEngine** (ингест, очистка, сортировка, FX‑коррекции)  
- **Memory‑aware pipeline** (chunking, mmap, Polars/Arrow)  
- **SnapshotEngine (v1)** — базовый детерминизм  

## Почему это важно:
Если время и данные не идеальны — всё остальное будет ложным.

## Критерии готовности:
- 1 млн тиков прогоняются **детерминированно**  
- одинаковый результат при повторном запуске  
- нет NaN, нет пропусков, нет рассинхронизации

TARE — Tick‑Level Algorithmic Research Environment.  
Это исследовательская среда для работы с биржевыми данными на уровне тиков, включающая:

чтение и очистку данных,

моделирование задержек,

воспроизведение исторических потоков,

создание симуляций,

анализ стратегий.

Проект должен быть детерминированным, воспроизводимым, модульным и расширяемым.

🏗 2. Архитектурные принципы
Строгая модульность  
Каждый модуль отвечает за одну чёткую задачу.

Детерминизм  
Никакой случайности, никакого float, только int и чистые функции.

Прозрачность данных  
Все преобразования должны быть объяснимы.

Минимум зависимостей  
Только стандартная библиотека Python.

Чёткая структура директорий  
Никаких дублирующих имён, никаких неоднозначных путей.

ИИ‑агент обязан предупреждать о рисках  
Если структура проекта выглядит подозрительно — агент сообщает.

🧠 4. Модули и их роли
4.1. time_engine/
Моделирует время в симуляции.

TimeEngine — преобразование market_time → local_time → simulation_time

Детерминированная задержка

Монотонный счётчик событий

4.2. tick_data_engine/
Работа с сырыми тиковыми данными.

tick_reader.py — чтение бинарных/CSV потоков

tick_cleaner.py — очистка, фильтрация

jitter_corrector.py — коррекция временных артефактов

4.3. snapshot/
Работа с историческими снимками.

snapshot_v1.py — структура snapshot

replay.py — воспроизведение исторических данных

4.4. memory/
Высокопроизводительные структуры данных.

ring_buffer.py — циклический буфер

chunking.py — разбиение данных на чанки

mmap_loader.py — memory‑mapped файлы

🧭 5. Правила для ИИ‑разработчика
5.1. Архитектурные проверки
ИИ обязан:

проверять структуру директорий,

предупреждать о дублирующих именах,

сообщать о потенциальных конфликтах,

не создавать файлы в подозрительных местах,

не генерировать код, если путь выглядит неправильным.

5.2. Генерация кода
ИИ должен:

генерировать полный файл, а не фрагменты;

добавлять docstring;

соблюдать стиль проекта;

не придумывать детали, если ТЗ неполное;

запрашивать уточнения.

5.3. Автодебаг
ИИ:

запускает файл,

ловит ошибку,

анализирует проект,

предлагает патч,

объясняет причину.

5.4. Автогенерация мелких модулей
Для файлов вроде chunking.py достаточно:

описания роли модуля,

интерфейса (методы, параметры),

ограничений.

ИИ выводит остальное сам.
